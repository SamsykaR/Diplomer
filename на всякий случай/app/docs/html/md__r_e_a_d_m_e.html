Пользовательская документация (RestN)

Обзор проекта

Название: RestN
Тип: кроссплатформенное приложение для планшетов (Android / Windows) на Qt 6
Назначение: автоматизация приёма заказов, управления складом, формирования отчётности, контроля активных заказов и генерации чеков в ресторане.

Ключевые возможности:

  • Авторизация сотрудников (официант, кладовщик, администратор)
  • Оформление заказов с автоматическим списанием ингредиентов
  • Возврат продуктов при удалении позиции из заказа
  • Редактирование существующих заказов (добавление/удаление позиций)
  • Управление складскими остатками (добавление, убавка, установка)
  • Отчёты по выручке, популярности блюд, дефицитным продуктам и чекам
  • Сохранение отчётов в CSV
  • Просмотр активных заказов, изменение статуса («Готовится» → «Готов» → «Закрыт»)
  • Формирование чека при закрытии заказа и его последующий просмотр
  • Ролевое разграничение доступа
  • Синхронизация данных по локальной сети (клиент-сервер)

Технологии программирования

При разработке использовались следующие инструменты и подходы:

  • Язык C++ – основная логика приложения, работа с сетью и базой данных.
  • Qt 6 – фреймворк для графического интерфейса (Qt Quick), сетевых сокетов и доступа к SQL.
  • PostgreSQL – централизованная СУБД для серверного режима.
  • CMake – система сборки проекта.
  • Qt Creator – интегрированная среда разработки.

Архитектурные подходы:
Клиент-серверная архитектура, паттерн Model-View (данные передаются через QVariantList), механизм сигналов и слотов для коммуникации между объектами, а также Singleton для классов Logs и Settings.

Описание программных модулей

Модуль работы с базой данных (DBWorker)

Класс DBWorker предназначен для выполнения всех операций с базой данных PostgreSQL. Открывает соединение в конструкторе, поддерживает переподключение через reconnect. Основные методы:

  • checkUser(login, password) – аутентификация с хешированием SHA3-256 + соль.
  • getFoods() – возвращает список блюд с доступностью (рассчитывается по остаткам) и составом.
  • getProducts() – список продуктов с остатками и минимальным порогом.
  • updateProductQuantity() / addProductQuantity() – изменение остатков.
  • checkAndConsume(foodId, quantity) – в транзакции проверяет наличие ингредиентов и списывает их.
  • returnIngredients(foodId, quantity) – возвращает ингредиенты на склад.
  • createOrder() / addOrderItem() – создание заказа.
  • Методы отчётов: getReportRevenue, getRevenueByDay, getFoodPopularity, getReportLowStock.
  • saveReportToCSV() – сохранение отчёта в CSV.
  • getOrders(status) – возвращает список заказов с фильтром по статусу.
  • updateOrderStatus(orderId, newStatus) – изменяет статус заказа.
  • getOrderItems(orderId) – возвращает позиции заказа.
  • closeOrder(orderId, totalSum, items) – закрывает заказ, формирует чек.
  • getCheckByOrderId(orderId) – получает данные чека.
  • syncOrderItems(orderId, newItems) – синхронизирует позиции заказа при редактировании.
  • getChecksByDateRange(from, to) – формирует список чеков за период.

Все публичные методы обрабатывают исключения (DatabaseEx, ValidateEx), логируют ошибки и испускают сигнал errorOccurred для QML.

Модуль сетевого клиента (TcpClient)

Класс TcpClient дублирует интерфейс DBWorker, но вместо прямых запросов к БД отправляет JSON-команды на сервер. Соединение устанавливается методом connectToServer(host, port). Протокол обмена: запрос/ответ в виде строк JSON, завершённых символом \n. Для синхронного ожидания используется waitForResponse с таймаутом 60 секунд. При потере соединения генерируется сигнал errorOccurred.

Пример запроса:

{"command": "checkUser", "requestId": 1, "data": {"login": "waiter1", "password": "123"}}

Ответ сервера:

{"response": true, "requestId": 1, "data": {"role": "waiter", "name": "Иванов", "userId": 5}}

Модуль сетевого сервера (TcpServer + ClientConnection)

TcpServer запускает серверный сокет и при подключении клиента создаёт объект ClientConnection. В ClientConnection::onReadyRead() данные накапливаются в буфере, извлекаются по разделителю \n, парсится JSON и вызывается соответствующий метод DBWorker. Результат отправляется обратно в виде JSON. Сервер способен параллельно обрабатывать несколько клиентов.

Модуль логирования (Logs)

Класс Logs записывает сообщения с уровнями INF, WARN, ERR в файл и дублирует их в консоль. При недоступности основного файла использует резервный путь в системной папке приложения. Формат строки: yyyy-MM-dd hh:mm:ss [УРОВЕНЬ] сообщение.

Модуль настроек (Settings)

Класс Settings предоставляет Q_PROPERTY для доступа из QML. Параметры (хост, порт БД, сервер, режим, пути к логам и отчётам) сохраняются через QSettings в системном реестре/конфигурационном файле. Методы load() и save() вызываются при запуске и при закрытии диалога настроек.

Пользовательский интерфейс (QML)

  • LoginPage.qml – экран входа. Поля «Логин», «Пароль», кнопки «Войти» и «Настройки». При успехе испускает сигнал loginSuccess.
  • Main.qml – главное окно. Панель инструментов с кнопками меню (☰) и синхронизации (⟳), выдвижное меню (Drawer), пункты которого зависят от роли пользователя.
  • OrderPage.qml – страница заказа. Левая колонка – меню с фильтрами, правая – корзина. Кнопки «Добавить», «Удалить», «Оформить заказ». Поддерживается режим редактирования (при передаче editMode и orderId).
  • StockPage.qml – страница склада. Список продуктов, поле ввода количества, кнопки «Добавить», «Убавить», «Установить». Продукты ниже минимального порога подсвечиваются.
  • ReportsPage.qml – страница отчётов. Выбор типа отчёта (выручка, популярность блюд, малые остатки, чеки), фильтры (даты, блюдо, столик, официант), кнопка «Сформировать», вывод результатов и «Сохранить отчёт» (CSV).
  • ActiveOrdersPage.qml – страница активных заказов. Отображает заказы, созданные сегодня, с фильтрацией по статусу. Для каждого заказа доступны действия: «Изменить» (открыть в режиме редактирования), «Готов» (изменить статус, только для администратора), «Оплатить» (закрыть заказ и сформировать чек), «Просмотр чека» (показать диалог с деталями).

Интеграция программных модулей

Подключаемые внешние модули

  • Qt 6 (Core, Quick, Sql, Network)
  • PostgreSQL client library (libpq) – для драйвера QPSQL.

Интеграция C++ и QML

Объекты регистрируются в контексте через setContextProperty():

engine.rootContext()->setContextProperty("dbWorker", worker);

В QML методы вызываются напрямую (dbWorker.checkUser(...)). Сигналы (например, errorOccurred) перехватываются через Connections.

Клиент-серверное взаимодействие

Используется TCP. Каждое сообщение – JSON-строка, заканчивающаяся \n. Клиент нумерует запросы (requestId) и ожидает соответствующий ответ. Таймаут – 60 секунд. Сервер обрабатывает каждого клиента в отдельном объекте, что позволяет параллельно обслуживать множество подключений.

Руководство пользователя

Начальный запуск

  1. Установите приложение на планшет (APK для Android или соберите под Windows).
  2. При первом запуске откроется окно авторизации.
  3. Если приложение работает в клиентском режиме, убедитесь, что сервер (другое устройство с программой в режиме сервера) активен и доступен по сети.
  4. При необходимости настройте подключение через кнопку «Настройки».

Окно авторизации

  • Поле «Логин» – введите имя пользователя.
  • Поле «Пароль» – введите пароль.
  • Кнопка «Войти» – проверка данных. При успехе открывается главное окно с соответствующими правами.
  • Кнопка «Настройки» – открывает диалог конфигурации.

Диалог настроек

Группа «База данных» (видна только в серверном режиме): хост, порт, имя БД, пользователь, пароль; кнопка «Проверить подключение».

Группа «Сеть»: IP сервера, порт; флажок «Режим сервера»; кнопки «Проверить сервер», «Показать IP устройства».

Группа «Прочее»: путь к логам, папка для отчётов.

Сохранение – кнопки «OK» (сохранить и закрыть) и «Отмена».

Главное окно и выдвижное меню

  • Верхняя панель: кнопка ☰ (открыть меню), название «Ресторан», кнопка ⟳ (синхронизация данных).
  • Выдвижное меню:
    • Отображается ФИО пользователя.
    • Пункты зависят от роли: официант – «Оформление заказа», «Активные заказы»; кладовщик – «Склад»; администратор – «Оформление заказа», «Склад», «Отчёты», «Активные заказы».
    • Кнопки: «Настройки», «Выйти», «Закрыть приложение».
  • Меню закрывается касанием вне его или повторным нажатием кнопки ☰.

Оформление заказа (страница для официанта)

Левая колонка – меню:

  • Фильтр по категории (Супы, Горячее, Салаты, Напитки, Все)
  • Флажок «Только доступные»
  • Список блюд: название, цена, категория, описание, состав. Недоступные блюда помечены красным.

Правая колонка – заказ:

  • «Текущий заказ» – корзина с добавленными блюдами.
  • Поля: «Столик №» (1–50), «Кол-во» (1–10), «Комментарий».
  • Кнопка «Добавить» – проверяет наличие ингредиентов. Если после списания остатки падают ниже порога, появляется предупреждение (добавление разрешается). При нехватке продуктов – ошибка.
  • Кнопка «Удалить» – удаляет позицию из корзины и возвращает ингредиенты.
  • Кнопка «Оформить заказ» – фиксирует заказ в БД (статус «Готовится»), очищает корзину и обновляет меню.

Режим редактирования: если страница открыта с параметром editMode: true и передан orderId, загружается существующий заказ. Кнопка «Оформить заказ» заменяется на «Сохранить изменения». При сохранении вызывается syncOrderItems, который корректно списывает и возвращает ингредиенты в зависимости от изменений.

Управление складом (страница для кладовщика)

  • Список продуктов: название, текущее количество (кг, л, шт.). Продукты ниже минимального порога подсвечены розовым.
  • Выбор продукта – касание строки.
  • Поле «Количество» (целое число 0–10000).
  • Кнопки: «Добавить» (увеличить остаток), «Убавить» (уменьшить, если результат ≥0), «Установить» (задать точное значение). Изменения немедленно синхронизируются с сервером.

Отчёты (страница для администратора)

Типы отчётов (выпадающий список):

  1. Выручка – фильтры: дата «с», «по», блюдо, столик, официант. Выводится общая выручка и разбивка по дням.
  2. Популярность блюд – фильтры: дата «с», «по». Результат – список блюд с количеством проданных порций.
  3. Малые остатки – без фильтров. Отображаются продукты, у которых остаток ниже порога.
  4. Чеки – фильтр по датам («с» и «по»). Отображается таблица с номерами чеков, суммой, номером столика, официантом и датой. Для каждого чека доступна кнопка «Просмотр» с полным содержимым.

Сохранение отчёта: после формирования нажмите «Сохранить отчёт», выберите имя файла. Файл будет создан в папке, указанной в настройках (по умолчанию reports). Формат – CSV (разделитель запятая, кодировка UTF-8).

Активные заказы (страница для официанта и администратора)

  • Отображаются заказы, созданные сегодня.
  • Фильтр по статусу: «Все», «Активный» (статус «Новый»), «Готовится», «Готов», «Закрыт».
  • Флажок «Скрыть закрытые» – убирает из списка заказы со статусом «Закрыт».

Для каждого заказа доступны действия:

  • «Изменить» – открывает страницу оформления заказа в режиме редактирования (для любых незакрытых заказов).
  • «Готов» – переводит заказ из статуса «Готовится» в «Готов» (доступно только администратору).
  • «Оплатить» – закрывает заказ (статус становится «Закрыт»), формирует чек и сохраняет его в базе данных (доступно для заказов со статусом «Готов»).
  • «Просмотр чека» – показывает сохранённый чек для закрытых заказов.

Страница автоматически обновляется при изменении статуса заказа или создании нового заказа.

Настройки и системные файлы

  • Файл настроек – хранится через QSettings (реестр Windows или конфигурационный файл). Вручную не редактировать.
  • Лог-файл – по умолчанию logs.txt в рабочей папке. При ошибке записи используется системная папка приложения.
  • Папка отчётов – задаётся в диалоге настроек, используется для сохранения CSV-файлов.

Сборка проекта (для разработчиков)

Требования:

  • Qt 6.8 или новее (Core, Quick, Sql, Network)
  • CMake ≥ 3.16
  • PostgreSQL (серверный режим) либо клиентская часть без БД.

Шаги сборки:

git clone <url>
cd RestN
mkdir build && cd build
cmake ..
cmake --build . --config Release

Запуск на Android: выберите в Qt Creator комплект Android, соберите APK, установите на планшет. На серверном устройстве должно быть установлено приложение (режим сервера) или отдельный PostgreSQL.
